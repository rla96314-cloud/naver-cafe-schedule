# 기획: 대문 자동 업로드 (마지막 수동 단계 제거)

2026-07-06. 기획만 — 구현 전 P0 스파이크 필수.
현재 루틴: 시트 수정 → 도구 열기(자동 동기화) → HTML 복사 → **[수동] 대문 편집에 붙여넣기 → 저장**.
이 마지막 수동 단계를 자동화할 수 있는가?

## 결론 요약

가능하다. 관건은 기술이 아니라 **"네이버 매니저 계정 로그인을 기계에 맡길 것인가"**.
로그인을 사람이 쥐고 있으면 안전하고(B안), 기계에 넘기면 완전 자동이지만 깨질 수 있다(C안).

| | A. 원클릭 주입 (추천 1순위) | B. 로컬 반자동 | C. 서버 완전 자동 |
|---|---|---|---|
| 방식 | 크롬 확장/북마클릿 | 내 PC 브라우저 자동화 | restream 서버 cron |
| 사람이 하는 일 | 대문 편집 페이지에서 버튼 1클릭 | 스크립트 1회 실행 | 없음 (매주 자동) |
| 로그인 | **내 브라우저 세션 그대로** (문제 없음) | 저장된 프로필 재사용 | 자동 로그인 필요 ⚠ |
| 깨질 위험 | 낮음 (에디터 DOM 변경 시만) | 중간 | 높음 (로그인 차단·캡차·기기확인) |
| 운영진 공유 | 확장 파일 전달로 가능 | 어려움 | 한 명 계정으로 전원 혜택 |
| 계정 리스크 | 없음 | 낮음 | 있음 (봇 감지 시 보호조치) |

## 왜 이 구조인가 — 제약 사실

- 네이버 카페는 **대문 편집 공개 API가 없다** (v1 설계문서에서 확정). 남은 길은
  ① 브라우저(에디터)를 조작하거나 ② 에디터가 쏘는 저장 요청을 흉내내는 것.
- 로그인 자동화가 유일한 진짜 리스크: 캡차·새 기기 확인·2단계 인증이 언제든 끼어든다.
  (참고: crew-collector에서 자동로그인 운영 경험 있음 — 되긴 하나 관리 비용이 있다는 것도 경험함)
- 우리는 이미 결과 검증 수단이 있다: 붙인 뒤 대문을 헤드리스로 열어 스크린샷 대조(이번 세션에서 실제로 함).

## A안 — 원클릭 주입 (크롬 확장) ★추천

**흐름**: 평소처럼 브라우저로 카페 관리 → 대문 편집 열기(로그인은 원래 세션) →
확장 버튼 클릭 → 확장이 시트에서 직접 이번 주 HTML을 생성해 에디터에 주입 + 저장 클릭.

- 도구(생성기)는 이미 순수 JS 모듈이라 **확장에 그대로 내장 가능** (generate.mjs 재사용).
  즉 "복사→붙여넣기"조차 사라지고, 확장이 시트에서 바로 만들어 넣는다.
- 로그인·캡차 문제가 **원천적으로 없음** — 사람이 이미 로그인된 페이지에서 실행되므로.
- 운영진 배포: 확장 폴더(개발자 모드 로드) 또는 크롬 웹스토어 비공개 배포.
- 깨지는 경우: 네이버가 대문 편집 페이지 DOM을 개편할 때만. 그땐 셀렉터 수정 1회.

**P0-A 스파이크 (30분)**: 대문 편집 페이지에서 개발자도구로
① HTML 모드 textarea/iframe의 DOM 경로 ② 값 주입 후 '저장' 버튼 셀렉터 ③ 주입이 에디터 상태에 반영되는지(React류면 네이티브 setter 필요) 확인.

## B안 — 로컬 반자동 (브라우저 자동화)

**흐름**: `node upload.mjs` 실행 → 저장된 크롬 프로필(1회 수동 로그인)로 Playwright가
대문 편집 열기 → HTML 교체 → 저장 → 스크린샷 검증 → 완료 알림.

- 로그인은 "저장된 세션 재사용"이라 평소엔 자동, 세션 만료 시만 창 띄워 사람이 로그인.
- 사실상 A안의 CLI 버전. A안 스파이크 결과를 그대로 재사용.
- 내 PC에서만 — 운영진 공유엔 부적합.

## C안 — 서버 완전 자동 (cron)

**흐름**: restream 서버(Tailscale, Docker — crew-collector와 동일 패턴)에서
매주 월 09:00 cron → 시트에서 이번 주 생성 → puppeteer 자동 로그인 → 대문 교체 → 저장 →
대문 스크린샷 검증 → 디스코드/카톡 알림(성공 스샷 or 실패 사유).

- 사람 개입 0. 시트만 채우면 월요일 아침 대문이 바뀌어 있음.
- **리스크**: 매니저 계정 자동 로그인 — 캡차/새 기기 확인에 걸리면 그 주는 실패 → 알림 받고 수동.
  로그인 세션 쿠키를 길게 유지(주 1회만 사용)하고, 실패 시 사람에게 넘기는 설계가 필수.
- 안전장치(필수): ① 교체 전 기존 대문 HTML 백업 저장 ② 저장 후 스크린샷 자동 대조
  ③ 실패 시 이전 상태 유지 + 알림 ④ 로그인 재시도 금지(1회 실패 = 사람 호출, 계정 보호).

## 숨은 4번째 길 — 저장 요청 직접 재현 (스파이크 가치 높음)

에디터의 '저장'이 단순 **HTTP POST 1방**이라면, 브라우저 자체가 필요 없다:
쿠키(로그인 세션)만 있으면 서버에서 `POST /...대문저장` 을 그대로 쏘면 끝.
- C안이 "puppeteer 없이 requests 몇 줄"로 줄어듦 — 가장 가볍고 견고한 완전 자동.
- **P0-D 스파이크**: 대문 저장 순간 개발자도구 Network 탭 캡처 → 요청 1개로 재현 가능한지,
  CSRF 토큰이 있는지(있으면 편집 페이지 GET에서 추출 가능한지) 확인.

## 권장 로드맵

1. **P0 스파이크 (1시간, 같은 세션에서 A와 D 동시 확인)**
   - 대문 편집 페이지 열고: 에디터 DOM 구조(A용) + 저장 시 네트워크 요청(D용) 캡처.
   - 이 결과가 구현 난이도를 확정한다. (지금까지의 교훈: 네이버는 추측 금지, 실측만.)
2. **1단계 = A안 확장** — 리스크 0으로 "1클릭 업로드" 달성. 운영진 배포.
3. **2단계 = D(가능하면) 또는 C** — 완전 자동이 정말 필요해지면. A가 이미 1클릭이라
   체감 이득은 "클릭 1번 vs 0번" — 서두를 필요 없음.

## 판단 기준 (언제 C까지 가나)

- 매주 월요일 아침에 사람이 아무도 시간이 안 될 때
- 대문 갱신 깜빡함이 실제로 자주 발생할 때
- 그 전까지는 A안(1클릭)이 리스크 대비 최적.
